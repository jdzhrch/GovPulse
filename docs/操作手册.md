# GovPulse 操作手册

> 面向政策团队和产品经理的使用指南

---

## 一、系统概述

### 1.1 GovPulse 是什么？

GovPulse 是一个 **政策监控与影响评估平台**，帮助团队：

- **发现**：自动扫描目标市场的最新法规动态
- **评估**：AI 分析政策对业务的具体影响
- **行动**：生成合规差距和产品整改建议

### 1.2 谁应该使用这个系统？

| 角色 | 使用场景 |
|------|----------|
| **Policy Analyst** | 发起扫描、查看政策详情、评估风险等级 |
| **Policy Lead** | 审核 AI 分析结果、确认优先级、推送给 PM |
| **Product Manager** | 查看被推送的合规需求、评估产品改动工作量 |

### 1.3 系统入口

- **线上版本**：https://jdzhrch.github.io/GovPulse
- **本地开发**：http://localhost:3000/GovPulse

## 二、核心概念

### 2.1 扫描任务（Mission）

一次扫描任务 = 选定市场 + 政策领域 + 时间范围

系统会搜索该组合下的政策动态，输出一份任务报告（Mission Report）。

### 2.2 影响评估（Impact Assessment）

对每条发现的政策信号，AI 会生成一份影响评估报告，包含：

| 字段 | 说明 |
|------|------|
| **Risk Level** | 风险等级（Critical / High / Medium / Low） |
| **Risk Rationale** | 为什么是这个风险等级 |
| **Business Impact** | 对业务的具体影响 |
| **Compliance Gaps** | 与现行做法的差距（需要改什么） |
| **Remediations** | 建议的整改措施（怎么改） |
| **Deadline** | 合规截止日期（如有） |

### 2.3 风险等级定义

| 等级 | 定义 | 响应时间 | 典型场景 |
|------|------|----------|----------|
| 🔴 **Critical** | 法规已生效或即将生效，直接影响核心业务 | 24-48 小时 | 禁令、高额罚款、必须下架 |
| 🟠 **High** | 法规确定会通过，需要产品改动 | 1-2 周 | 新增年龄验证、数据本地化 |
| 🟡 **Medium** | 草案阶段，可能影响业务 | 1-3 个月 | 正在审议的提案 |
| 🟢 **Low** | 早期讨论或影响有限 | 季度回顾 | 行业倡议、其他市场参考 |

### 2.4 支持的市场和领域

**市场（Market）**：
- 🇧🇷 Brazil | 🇰🇷 South Korea | 🇲🇽 Mexico | 🇵🇭 Philippines
- （可通过代码扩展更多市场）

**政策领域（Domain）**：
| 图标 | 领域 | 说明 |
|------|------|------|
| 🌐 | All Domains | 扫描全部领域 |
| 🛡️ | Youth Safety | 未成年人保护 |
| 🛒 | Digital Commerce | 电商、支付、直播打赏 |
| 🗄️ | Data & Privacy | 数据保护、本地化、跨境传输 |
| 💬 | Content Policy | 内容审核、有害信息 |

## 三、操作指南

### 3.1 首页仪表盘（Dashboard）

进入系统后的默认页面，展示：

**统计卡片**：
- **Critical Alerts**：需要立即关注的高风险政策数量
- **Policies Tracked**：已扫描的政策信号总数
- **Markets Monitored**：覆盖的市场数量
- **Pending Actions**：待处理的合规差距数量

**最近评估列表**：
- 按风险等级排序的影响评估报告
- 点击可进入详情页

### 3.2 发起新扫描（New Policy Scan）

点击顶部导航的 **「New Scan」**

#### 操作步骤：

1. **填写扫描背景（可选）**
   - 描述触发这次扫描的原因
   - 例如："Brazil 青少年保护新规进展"

2. **选择目标市场**
   - 点击国旗卡片选择一个市场
   - 每次扫描只能选一个市场（确保分析深度）

3. **选择政策领域**
   - 默认"All Domains"覆盖全部领域
   - 也可指定某个特定领域聚焦扫描

4. **选择回溯时长**
   - 30 天：近期动态（快速扫描）
   - 90 天：季度回顾（推荐）
   - 180 天：半年盘点
   - 365 天：年度全景

5. **点击「Start Live Scan」**
   - 系统会触发 GitHub Actions 执行扫描
   - 进度条显示当前状态
   - 完成后自动跳转到报告列表

#### 扫描过程说明：

| 阶段 | 耗时 | 说明 |
|------|------|------|
| Queued | ~10s | 等待 GitHub Actions 启动 |
| Scout Engine | ~30s | 搜索政策来源，发现信号 |
| Impact Analyzer | ~60s | AI 分析每条信号的影响 |
| Deploy Results | ~20s | 保存结果并更新页面 |

**总耗时约 2-3 分钟**，请耐心等待。

### 3.3 查看扫描报告（Scan Reports）

点击顶部导航的 **「Scan Reports」**

#### 报告列表：
- 显示所有历史扫描任务
- 支持按市场、日期范围筛选
- 点击任意任务查看详情

#### 任务详情页：
- **任务摘要**：市场、领域、扫描时间
- **发现的政策列表**：该任务下所有政策信号
- **风险分布**：按等级统计

### 3.4 查看政策详情（Policy Details）

点击顶部导航的 **「Policy Details」**

#### 列表页面：
- 展示所有影响评估报告
- 支持筛选：
  - **按市场**：选择特定国家
  - **按风险等级**：只看 Critical/High
  - **按时间**：最近 7 天 / 30 天 / 90 天

#### 详情页面：

**顶部信息栏**：
- 政策标题
- 风险等级标签
- 来源市场
- 分析时间

**三栏摘要**：
| 风险原因 | 业务影响 | 合规截止日期 |
|----------|----------|--------------|
| 为什么是这个风险等级 | 对业务的具体影响 | 需要在何时完成合规 |

**合规差距（Compliance Gaps）**：
- 列出每条需要关注的差距
- 显示严重程度（Critical / Major / Minor）
- 说明具体内容

**整改建议（Remediations）**：
- 针对每条差距的建议措施
- 显示优先级（P0 / P1 / P2）
- 预估工作量（工程天数）
- 建议负责团队

### 3.5 推送给 PM

在政策详情页，点击右上角的 **「Mark as Reviewed」** 按钮。

- 表示 Policy Lead 已确认此评估
- 按钮变为绿色 ✅「Reviewed」并显示时间
- 此状态可用于追踪哪些政策已被处理

## 四、最佳实践

### 4.1 扫描频率建议

| 市场类型 | 建议频率 | 理由 |
|----------|----------|------|
| Tier 1（重点市场） | 每周 | 政策变化快，影响大 |
| Tier 2（次重点） | 每两周 | 保持关注但不需高频 |
| Tier 3（观察市场） | 每月 | 了解趋势即可 |

### 4.2 风险响应流程

```
发现 Critical 政策
       ↓
Policy Lead 确认（24h 内）
       ↓
拉齐相关 PM 和法务
       ↓
制定响应方案（48h 内）
       ↓
启动产品改动
```

### 4.3 与现有工作流集成

**建议方式**：
1. 每周一发起 Tier 1 市场扫描
2. 周二 Policy Lead review AI 分析结果
3. 确认后在周三 sync 会上和 PM 对齐
4. High/Critical 项进入产品 backlog

## 五、技术说明

### 5.1 数据来源

- **Scout Engine** 使用 Tavily API 搜索：
  - 政府官网公告
  - 权威新闻媒体
  - 行业分析报告

### 5.2 AI 分析原理

- 使用 GPT-4o 进行影响评估
- 对比 `knowledge/internal_baseline.md` 中的公司现状
- 输出结构化的 JSON 格式便于系统处理

### 5.3 数据存储

- 所有扫描结果保存在 `data/history/` 目录
- 文件格式：
  - `MISSION-{市场}-{时间戳}.json`：任务元数据
  - `IMPACT-{ID}.json`：影响评估详情
  - `SUMMARY-{任务ID}.md`：可读摘要

### 5.4 内部基准维护

文件位置：`knowledge/internal_baseline.md`

包含内容：
- 当前产品功能现状（年龄门控、数据存储等）
- 已有合规措施
- 组织架构和责任人

**更新方式**：由 Policy 团队定期 review，通过 Git 提交更新。

## 六、常见问题

### Q: 扫描卡在"Running"很久怎么办？
A: GitHub Actions 执行可能需要 2-3 分钟，请耐心等待。如超过 5 分钟，可刷新页面或检查 GitHub Actions 页面。

### Q: AI 分析结果不准确怎么办？
A: AI 输出是建议，不是决策。Policy Lead 应 review 后再确认。如发现系统性偏差，可以反馈给团队优化 prompt。

### Q: 如何添加新的市场？
A: 需要在代码中配置，联系工程团队。需要提供：市场代码、显示名称、主要政策来源。

### Q: 历史数据会保留多久？
A: 所有数据永久保存在 Git 仓库中，可随时追溯。

### Q: 可以批量扫描多个市场吗？
A: 当前版本每次扫描一个市场。如需批量，可多次发起或等待后续版本支持。
